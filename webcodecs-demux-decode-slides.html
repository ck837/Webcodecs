从 MP4 文件到 Canvas 像素 —— 容器、码流、解码器各管什么

解封装 Demux VideoDecoder H.264 / avc1 DJI 图传场景
01 · 核心问题

VideoDecoder 不能直接吃 MP4 文件

磁盘上的 .mp4 是一个容器，里面混着索引、元数据、多路轨道。硬解码器只认纯 H.264 压缩比特流。

解封装 = 拆快递盒；decode = 按说明书把压缩帧还原成像素。

MP4 文件 → 解封装 → EncodedVideoChunk → decode() → VideoFrame 像素
02 · 两层结构

容器 vs 码流

层级是什么谁处理
容器MP4 / M3U8+TS / MKVmp4box、mux.js
码流H.264 NAL 单元 + SPS/PPSVideoDecoder

容器负责：打包、索引、时间戳、多轨同步
码流负责：实际的视频压缩数据

03 · 背景：为什么要编码

视频在文件里本来就是「压缩过的」

原始 1080p@30fps 未压缩视频 ≈ 745 Mbps，1 分钟 ≈ 5.5 GB。编码（H.264）把它压到几 Mbps，才能存储和传输。

格式1 分钟 1080p 大约用途
原始 YUV 像素~5 GB相机传感器、GPU 纹理
H.264 编码文件~30–150 MBMP4、直播、图传

你的 Demo 做的是播放侧：文件里已是 H.264，只负责解封装 + 解码 + 显示，不负责编码。

04 · MP4 里面有什么

为什么不能整文件丢给 decode？

  • ftyp — 文件类型标识，不是视频帧
  • moov — 元数据：轨道、编解码参数
  • mdat — 压缩帧二进制，但边界靠索引表
  • 可能还有音频轨、字幕轨……

解码器不知道：哪段是视频、每帧从哪切、SPS/PPS 在哪、时间戳是多少。

05 · 解封装产出两样东西

解封装 ≠ 解码，它只做「拆包 + 分拣」

① description（来自 avcC）

SPS / PPS / profile / level
→ 交给 VideoDecoder.configure()
解码器开机前的「说明书」

② EncodedVideoChunk（逐帧 sample）

纯 H.264 NAL + timestamp + key/delta
→ 交给 decoder.decode(chunk)
每一帧的「压缩正文」

③ 是说明书，⑤ 才是正文 —— 没有 avcC，decode 会失败或花屏。

从 MP4 到 Canvas 的 9 步

07 · configure 与 decode 的分工

两步 API，两种输入

API输入来自解封装的哪一步作用
decoder.configure() codec: 'avc1…'
description: Uint8Array		 ③ avcC（SPS/PPS） 初始化硬解码器，告诉它分辨率、profile 等
decoder.decode() EncodedVideoChunk ⑤ 每帧 sample 送入一帧压缩数据，异步产出 VideoFrame

顺序不能乱：必须先 configure，再 decode；key 帧通常要先于 delta 帧。

08 · 错误 vs 正确

如果跳过解封装会怎样？

❌ 错误做法

fetch('video.mp4') → 整文件 bytes → decoder.decode()

  • 找不到帧边界
  • 没有 SPS/PPS → configure 失败
  • 音频/索引数据混入 → 解码器报错
  • 没有时间戳 → 无法同步播放

✓ 正确做法

mp4box 解封装 → avcC + samples → configure → decode

  • 精确的每帧 NAL 边界
  • description 来自 avcC box
  • 只送视频轨压缩数据
  • microseconds 时间戳来自 stts
09 · M3U8 / TS 路径

容器不同，逻辑相同

TS 是另一种容器（188 字节包），mp4box 不能直接读。需要先用 mux.js transmux → fMP4，之后复用同一套 avcC + sample 提取。

.m3u8 文本 → 下载 .ts 分片 → mux.js transmux → fMP4 init + media → 与 MP4 相同 ③~⑨

三条路径（MP4 / M3U8 / TS）在 第 ④ 步 configure 处汇合。

10 · 和 <video> 的关系

浏览器内置播放器也在做同样的事

<video src="a.mp4"> 看起来一步播放，内部流程：

内置 Demuxer（解封装） ↓ 内置 Decoder（解码） ↓ Compositor（渲染到屏幕）

WebCodecs 的区别

把 demuxer 和 decoder 拆开给你控：
Canvas 绘制、帧级处理、低延迟图传、AI 逐帧分析……

代价：解封装要自己做（mp4box / mux.js）。

11 · 裸 H.264 流（图传场景）

大疆图传：可能没有 MP4 容器

WebSocket / RTSP 传来的往往是裸 NAL 单元流，可以跳过 mp4box，但仍需：

  • 从首个 IDR 帧前解析 SPS/PPS → configure
  • 按 NAL 边界切分 → 手动构造 EncodedVideoChunk
  • 时间戳由传输协议或接收时刻提供
无人机 H.264 编码 → 无线传输 → 浏览器 decode → Canvas / WebGL

容器可以没有，configure + decode 不能省。

12 · 总结

三句话记住

  1. 解封装 — 从容器里拆出「说明书 (avcC)」和「逐帧压缩数据 (sample)」
  2. decode — 只认纯 H.264 比特流，不认 MP4 Box 结构
  3. VideoDecoder 是解码器，不是播放器；播放器 = 解封装 + 解码 + 渲染
